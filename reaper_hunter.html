Overview

Cos'è Reaper Hunter

Reaper Hunter è il mio progetto di informatica: uno sparatutto in prima persona sviluppato completamente in Unity 6 usando C#. L'idea era semplice — un gioco a ondate dove il player affronta nemici sempre più numerosi e pericolosi.

Ho scelto lo stile grafico PSX anni '90 ispirandomi a DOOM e ai primi Resident Evil: texture a bassa risoluzione, poligoni spigolosi, fog volumetrica e post-processing che emulano i limiti tecnici della PlayStation 1. Non una limitazione estetica — una scelta di stile.

Il gioco non è complicato. Ma costruirlo ha richiesto di applicare davvero i concetti di informatica: programmazione ad oggetti, algoritmi, salvataggio su file e ottimizzazione. Questo sito racconta come ci sono arrivato.

Settimana per settimana: decisioni, problemi incontrati e come li ho risolti.

Settimana 01 Setup

Progetto Unity e struttura iniziale

Ho creato il progetto su Unity 6 usando il template Universal Render Pipeline. La scelta URP è stata deliberata: volevo il controllo sul post-processing per replicare l'estetica PSX. Ho impostato la cartella degli script, importato la mappa e configurato il player base.

Il primo approccio al progetto è stato abbastanza caotico — mettevo script su script senza pensare all'architettura. Poi ho capito che avrei dovuto pianificare prima la struttura OOP.

Settimana 02 Movimento · Camera

Movimento FPS e MouseLook

Il PlayerController gestisce il movimento WASD con il CharacterController di Unity, mentre il MouseLook ruota la camera in base al mouse. Il cursore viene bloccato al centro dello schermo durante il gioco.

PlayerController.cs C# · Movimento 
public class PlayerController : MonoBehaviour {
    private CharacterController controller;
    public float speed = 6f;
    public float gravity = -9.81f;
    private Vector3 velocity;

    // Update viene chiamato ogni frame
    void Update() {
        float x = Input.GetAxis("Horizontal");
        float z = Input.GetAxis("Vertical");

        Vector3 move = transform.right * x + transform.forward * z;
        controller.Move(move * speed * Time.deltaTime);

        // Gravità simulata manualmente
        velocity.y += gravity * Time.deltaTime;
        controller.Move(velocity * Time.deltaTime);
    }
}
Test movimento player in scena
⚠ Problema
Il player cadeva attraverso il pavimento al primo playtest. La gravità non funzionava correttamente.
✓ Soluzione
Aggiunto il reset di velocity.y quando isGrounded == true. Senza il reset, la velocità verticale si accumulava negativamente anche a terra.
Settimana 03 OOP · Architettura

Struttura OOP: interfacce e classi astratte

Questa è stata la parte più importante del progetto. Ho progettato la gerarchia delle classi applicando i principi OOP: interfaccia IDamageable come contratto, classe astratta EntitaBase come base comune, e due sottoclassi concrete: PlayerStats e NemicoAI.

IDamageable.cs — Interfaccia Astrazione 
// Contratto: qualsiasi entità che può subire danno DEVE avere questi metodi
public interface IDamageable {
    float CurrentHealth { get; }
    float MaxHealth     { get; }
    bool  IsDead        { get; }

    void TakeDamage(float amount);
    void Die();
}
EntitaBase.cs — Classe Astratta Ereditarietà 
// Classe astratta: implementa IDamageable, codice comune a player e nemici
public abstract class EntitaBase : MonoBehaviour, IDamageable {

    [SerializeField] protected float maxHealth = 100f;
    protected float currentHealth;
    protected bool  isDead = false;

    // Proprietà pubbliche — incapsulamento
    public float CurrentHealth => currentHealth;
    public float MaxHealth     => maxHealth;
    public bool  IsDead        => isDead;

    // Logica comune scritta UNA sola volta (DRY)
    public virtual void TakeDamage(float amount) {
        if (isDead) return;
        currentHealth = Mathf.Clamp(currentHealth - amount, 0f, maxHealth);
        OnDamageReceived(amount);
        if (currentHealth <= 0f) Die();
    }

    // Hook astratti — ogni sottoclasse decide cosa succede
    protected abstract void OnDamageReceived(float amount);
    protected abstract void OnDeath();
}
💡 Perché è utile
Il WeaponController quando spara chiama TakeDamage() su qualunque oggetto IDamageable. Non sa se sta colpendo il player o un nemico — funziona grazie al polimorfismo. Se in futuro aggiungo un terzo tipo di entità, non tocco il codice di sparo.
Settimana 04 Armi · Sparo

Sistema di sparo con Raycast

Il sistema di sparo usa il Raycast hitscan: viene lanciato un raggio invisibile dal centro dello schermo verso la scena. Se tocca qualcosa con IDamageable, applica danno.

Le statistiche di ogni arma sono in un ScriptableObject WeaponData — un asset separato dal codice. Aggiungere una nuova arma significa creare un nuovo asset, senza toccare nessuno script.

WeaponController.cs — sparo hitscan Raycast · Polimorfismo 
private void SparaRaycast() {
    // Raggio dal centro esatto dello schermo
    Ray ray = playerCamera.ScreenPointToRay(
        new Vector3(Screen.width / 2f, Screen.height / 2f)
    );

    if (Physics.Raycast(ray, out RaycastHit hit, ArmaCorrente.portata)) {

        // Polimorfismo: non importa chi è colpito
        hit.collider.GetComponentInParent<IDamageable>()
            ?.TakeDamage(ArmaCorrente.danno);

        // Effetto visivo impatto
        if (ArmaCorrente.prefabImpatto != null) {
            var imp = Instantiate(ArmaCorrente.prefabImpatto,
                hit.point, Quaternion.LookRotation(hit.normal));
            Destroy(imp, 2f);
        }
    }
}

Settimana 05 AI · NavMesh

Nemici con NavMesh e tre tipi distinti

I nemici usano il NavMeshAgent di Unity per il pathfinding. Unity precalcola una "mappa di navigazione" (NavMesh) sulla geometria della scena, e i nemici la usano per trovare il percorso verso il player evitando ostacoli.

Ho creato tre varianti con un enum TipoNemico che modifica i valori base nel metodo Awake.

NemicoAI.cs — Logica di inseguimento NavMesh · Ereditarietà 
public class NemicoAI : EntitaBase {

    public enum TipoNemico { Normale, Veloce, Tank }
    [SerializeField] private TipoNemico tipo;

    void Update() {
        float dist = Vector3.Distance(transform.position, player.position);

        if      (dist <= rangeAttacco) { agent.isStopped = true;  TentaAttacco(); }
        else if (dist <= rangeVista)   { agent.isStopped = false; agent.SetDestination(player.position); }
        else                           { agent.isStopped = true; }
    }

    // Override del metodo astratto di EntitaBase
    protected override void OnDeath() {
        PlayerStats.Instance?.AddScore(punteggio);
        WaveManager.Instance?.NemicoMorto();
        Destroy(gameObject, 3f);
    }
}

Settimana 06 Ondate · Ottimizzazione

WaveManager e Object Pooling

Il WaveManager gestisce le ondate. Ogni ondata ha un numero di nemici, quanti possono essere in scena contemporaneamente e l'intervallo di spawn. Dopo le ondate configurate, genera ondate procedurali infinite con difficoltà crescente.

Per ottimizzare ho usato l'Object Pooling: invece di creare e distruggere nemici continuamente (costoso per il Garbage Collector), li precario in una coda e li riutilizzo.

WaveManager.cs — Object Pool Ottimizzazione · Queue 
// Pool: un dizionario di code per tipo di nemico
private Dictionary<string, Queue<GameObject>> pool = new();

void Awake() {
    // Precario gli oggetti all'avvio — una volta sola
    PrecaricaPool("Normale", prefabNormale, 10);
    PrecaricaPool("Veloce",  prefabVeloce,  5);
    PrecaricaPool("Tank",    prefabTank,    3);
}

// Prende un nemico dalla coda: O(1), nessuna allocazione
private GameObject PrendiDalPool(string tipo) {
    if (pool[tipo].Count > 0) {
        var obj = pool[tipo].Dequeue();
        obj.SetActive(true);
        return obj;
    }
    return Instantiate(prefab); // fallback se pool è vuoto
}

// Restituisce al pool invece di Destroy: O(1)
public void RitornaAlPool(string tipo, GameObject go) {
    go.SetActive(false);
    pool[tipo].Enqueue(go);
}
⚠ Problema
Alle ondate 5+ il gioco laggava durante lo spawn. Instantiate() chiamato molte volte in pochi frame causava picchi di CPU visibili nel Profiler.
✓ Soluzione
Con il pool, la complessità dello spawn diventa O(1). I nemici esistono già — vengono solo riattivati. Il lag è sparito completamente.

Settimana 07 Salvataggio JSON

Save System con System.IO

Ho implementato il salvataggio persistente con JSON. Il SaveSystem è una classe statica utility: non ha istanze, espone solo metodi statici. Usa JsonUtility di Unity per serializzare i dati e System.IO.File per scrivere su disco.

SaveSystem.cs — Scrittura e lettura JSON System.IO · Serializzazione 
[System.Serializable]
public class DatiPartita {
    public int    punteggio;
    public int    ondataRaggiunta;
    public int    nemiciUccisi;
    public float  tempoGiocato;
    public string dataOra;
}

public static void SalvaPartita() {
    var data = new DatiPartita {
        punteggio       = PlayerStats.Instance.Score,
        ondataRaggiunta = WaveManager.Instance.OndataCorrente,
        nemiciUccisi    = PlayerStats.Instance.KillCount,
        tempoGiocato    = Time.timeSinceLevelLoad,
        dataOra         = DateTime.Now.ToString("dd/MM/yyyy HH:mm")
    };
    // Serializza in JSON e scrive su file
    string json = JsonUtility.ToJson(data, prettyPrint: true);
    File.WriteAllText(SavePath, json);
}

Come funziona il gioco

Le principali meccaniche implementate e come sono state realizzate tecnicamente.

01
Sparo Hitscan

Raycast istantaneo dal centro camera. Nessun proiettile fisico — il colpo arriva subito. Rileva IDamageable e applica danno via polimorfismo.

Physics.Raycast(ray, hit, portata)
→ IDamageable.TakeDamage(danno)
02
Ricarica Coroutine

La ricarica usa una Coroutine: attende N secondi, poi aggiorna le munizioni. Nel frattempo blocca altri spari con un flag booleano.

IEnumerator Ricarica()
yield return WaitForSeconds(t)
03
AI con NavMesh

Pathfinding A* precalcolato. Tre stati: pattuglia (fermo), inseguimento (SetDestination), attacco (TakeDamage sul player).

agent.SetDestination(player.position)
← aggiornato ogni frame
04
Ondate Infinite

Ondate configurate manualmente più generazione procedurale. Difficoltà crescente: più nemici, più veloci, meno pausa tra spawn.

nemiciTotali = 5 + idx * 3
intervalloSpawn = max(0.5, 2 - idx*0.1)
05
Object Pooling

Pool di nemici pre-istanziati. SetActive(true/false) invece di Instantiate/Destroy. Eliminato completamente il lag durante lo spawn.

Queue<GameObject> pool
Dequeue() → O(1) spawn
06
Salvataggio JSON

Salvataggio automatico a fine ondata. Dati serializzati con JsonUtility e scritti su file con System.IO. Lettura all'avvio per mostrare il record.

JsonUtility.ToJson(data)
File.WriteAllText(path, json)
07
Cambio Armi

Lista di WeaponData ScriptableObject. Scrollwheel o tasti 1-2 per cambiare. Ogni arma ha danno, cadenza, caricatore e suoni separati.

List<WeaponData> armiData
indiceArma = (i+dir) % count
08
Sistema Vita OOP

IDamageable + EntitaBase gestisce vita per player e nemici con codice condiviso. TakeDamage, Die, Heal implementati una volta sola.

EntitaBase.TakeDamage()
→ abstract OnDeath()
09
Post-Processing PSX

Volume URP con Fog, Film Grain, Vignette e Color Adjustments. Shader retro low-poly per texture pixelate tipiche della PlayStation 1.

Riflessioni

Cosa ho imparato

I takeaway più importanti di questo progetto.

LEZIONE 01

Pianifica l'OOP prima di scrivere codice

Ho riscritto tre volte i sistemi vita e punteggio perché non avevo pianificato la gerarchia. Quando ho disegnato su carta la struttura IDamageable → EntitaBase → sottoclassi, tutto è diventato molto più chiaro.

Takeaway: 20 minuti di progettazione OOP risparmiano ore di refactoring.

LEZIONE 02

Il Safe Mode non è un problema — è un aiuto

Inizialmente mi spaventava vedere Unity in Safe Mode. Poi ho capito che la console mostra esattamente dove è l'errore e quale classe manca. Leggere il messaggio invece di farsi prendere dal panico è tutto.

Takeaway: L'errore ti dice sempre cosa fare — bisogna solo leggerlo.

LEZIONE 03

Object Pooling fa una differenza enorme

Il lag durante le ondate sembrava impossibile da risolvere. Poi ho capito che Instantiate() è costoso. Con il pool, i nemici esistono già — vengono solo riattivati. Il gioco è diventato fluido immediatamente.

Takeaway: L'ottimizzazione non è optional — è parte del design.

LEZIONE 04

ScriptableObject separa dati da codice

Mettere le statistiche delle armi in uno ScriptableObject è stata una delle decisioni migliori. Modificare il danno della pistola non richiede di aprire un file C# — basta cambiare il valore nell'asset.

Takeaway: Dati configurabili fuori dal codice = progetto molto più flessibile.

LEZIONE 05

Il JSON è semplice e potente

Implementare il salvataggio sembrava complicato. In realtà con JsonUtility.ToJson() e File.WriteAllText() bastano 5 righe. Il file è leggibile anche con un editor di testo.

Takeaway: Lettura e scrittura su file sono molto più accessibili di quanto sembri.

LEZIONE 06

Lo stile visivo semplifica tutto

Scegliere lo stile PSX low-poly ha eliminato metà dei problemi grafici. Non servivano texture ad alta risoluzione, animazioni complesse o modelli dettagliati — la semplicità era l'obiettivo.

Takeaway: Una scelta estetica coerente guida le decisioni tecniche e le semplifica.
